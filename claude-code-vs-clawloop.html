Claude Code vs ClawLoop

两种截然不同的 AI 编程工具架构：一个是 Anthropic 的重型终端 IDE，另一个是基于 OpenClaw Gateway 的轻量桌面客户端。本文基于 2026-03-31 泄露的 Claude Code 源码与 ClawLoop 当前代码进行对比。

Anthropic
Claude Code
住在终端里的 Agentic 编程工具。直接调用 Anthropic API，在本地执行文件读写、Shell 命令、搜索等全部操作。
源文件 ~1,900 代码行 512K+ 运行时 Bun UI React + Ink
OpenClaw 生态
ClawLoop
面向 OpenClaw Gateway 的桌面客户端。自身不调用 LLM，通过 WebSocket RPC 将所有推理与工具执行委托给 Gateway 侧的 Agent。
架构 Electron UI React + Tailwind 通信 WebSocket RPC 定位 薄客户端

核心定位差异

两个项目解决的根本问题不同，这决定了它们在架构上走向完全不同的方向。

Claude Code ClawLoop
产品形态 终端 CLI + TUI Electron 桌面应用
核心角色 全栈 Agent — 推理 + 执行一体 会话编排层 — 只管 UI 和任务管理
LLM 调用 本进程内直连 Anthropic API 不直接调用；委托给 OpenClaw Gateway
工具执行 本地 ~40 个内建工具 + MCP 由 Gateway 背后的 Agent 执行
目标用户 开发者 — 在终端中编码 运营/非技术用户 — 管理定时任务和自动化

架构模型

最本质的差异：Claude Code 是单体厚客户端，ClawLoop 是分布式薄客户端。

┌──────────────────────────────────────────────────────────────────┐ │ Claude Code（单体） │ │ │ │ 用户输入 → Commander CLI → QueryEngine → Anthropic API │ │ ↕ │ │ query.ts 循环 │ │ ↙ ↓ ↘ │ │ BashTool FileEdit GrepTool ... │ │ ↘ ↓ ↙ │ │ 结果写回消息列表 → 再次请求 │ └──────────────────────────────────────────────────────────────────┘ ┌──────────────────────────────────────────────────────────────────┐ │ ClawLoop（客户端-网关分离） │ │ │ │ 用户输入 → React UI → IPC → Electron Main → WebSocket RPC │ │ ↕ │ │ OpenClaw Gateway │ │ ↙ ↘ │ │ Author Agent Runner Agent │ │ (设计任务) (执行任务) │ │ │ │ 流式事件 ← event:chat ← Gateway ← Agent LLM 推理 + 工具执行 │ └──────────────────────────────────────────────────────────────────┘

推理循环

两者都实现了"思考-行动"循环，但承载方式截然不同。

Claude Code — 本地 Agentic Loop

  1. 拼装 System Prompt（默认模板 + CLAUDE.md + git 状态 + 协调器文案）
  2. QueryEngine.submitMessage() 维护消息列表，调 query.ts
  3. query.ts 调 Anthropic API 流式拉 assistant 内容
  4. 解析 tool_use 块 → canUseTool 权限检查 → 本地执行工具
  5. 工具结果写回消息列表 → 如果模型未 stop，回到第 3 步
  6. 可选：token 超预算时 compact 压缩上下文

ClawLoop — Gateway 代理模式

  1. 用户消息通过 chat.send RPC 发给 Gateway
  2. Gateway 将消息路由到 Author/Runner Agent
  3. Agent 在服务端完成 LLM 推理 + 工具执行的完整循环
  4. 每一步通过 event:chat 流式推送到客户端
  5. 客户端只负责渲染 assistant 内容和工具调用气泡
关键区别：Claude Code 的整个 agent 循环（推理 → 工具 → 推理 → ...）全在用户终端进程内完成；ClawLoop 的客户端完全不参与这个循环，只是一个"观众"。

工具系统

Claude Code 内建了一套庞大的工具框架；ClawLoop 则把工具能力完全交给了 Gateway。

Claude Code ClawLoop
工具数量 ~40 个内建工具 + MCP 扩展 客户端 1 个（task_card）；通用工具在 Gateway 侧
工具抽象 Tool<I, O, P> — Zod Schema + 权限模型 + 进度回调 无统一抽象；插件 JS 直接导出 description 和 handler
注册方式 getAllBaseTools() + feature flag 条件拼接 + MCP 合并 Gateway 侧 Agent 预配置；客户端通过 AGENTS.md 描述
延迟加载 shouldDefer + ToolSearchTool 按需发现 不涉及（客户端不管工具加载）
扩展协议 MCP（Model Context Protocol）原生支持 通过 Gateway 插件系统

权限模型

Claude Code ClawLoop
粒度 每次工具调用都经过权限判定 连接级：token + 设备签名 + scope
决策路径 规则引擎 → allow / deny / ask 分支 → 交互式 UI / Bridge 回调 Gateway 握手时校验身份，之后信任
模式 default / plan / bypass / auto / coordinator 等多种 单一 operator.admin scope
记忆 可持久化 always-allow/deny 规则 无客户端侧权限记忆
Claude Code 的权限系统是它最复杂的横切关注点之一。useCanUseTool 这个 React Hook 串联了规则引擎、分类器、交互式 UI、Bridge 远程审批、协调器工人等多条路径。ClawLoop 把这个复杂度完全转嫁给了 Gateway 侧。

复杂度分布

两个项目把复杂度放在了完全不同的地方。

Claude Code 的复杂度集中在

QueryEngine 46K 行 Tool 抽象 29K 行 权限多路径判定 MCP / LSP 集成 Bridge 双向通信 Coordinator 编排 Prompt 多层拼装 Token 预算与压缩

ClawLoop 的复杂度集中在

任务文件系统管理 Gateway 连接与重连 Agent 工作区种子 会话连续性保持 Cron / 运行记录 产物 HTTP 预览

总结

Claude Code 和 ClawLoop 代表了 AI 编程工具的两种范式：

Claude Code 是厚客户端单体——1,900 个文件、512K 行代码把 LLM 推理、工具执行、权限控制、IDE 集成、多 Agent 协调全部塞进一个终端进程。它的优势是极低延迟和完整的本地控制，代价是巨大的代码复杂度。

ClawLoop 是薄客户端 + 远程 Gateway——客户端只做会话 UI 和任务管理，所有智能层下沉到 OpenClaw Gateway。它的优势是客户端极度简洁、Agent 可独立升级，代价是依赖网络和 Gateway 的可用性。

两者不是互相替代的关系，而是面向不同场景的不同取舍：一个给在终端中深度编码的开发者，一个给需要管理自动化任务流的运营用户。
